Determinism status (batch creation / chunk→batch association)

For JSONL streaming, batch membership is deterministic given identical input bytes and the same --batch-size, --max-rows, --shard-id/--num-shards. The code path is:

File discovery is sorted (BatchProcessor.list_input_files() sorts glob results and S3 listings).
File-level sharding is deterministic (xxhash of path in BatchProcessor.shard_files()).
Within a file, rows are streamed in line order (stream_chunks_from_jsonl()), and row-level sharding (when a single huge file is used) is deterministic via xxhash.xxh64() of the chunk_id (or uid/guid/id fallback).
Batches are just fixed-size consecutive groups of emitted rows (BatchProcessor.batch_iterator()), so repeated runs produce the same chunk_id sequence per batch.
Important nuance: batch boundaries are order-based, not hash-bucket-based. If you insert/delete a record earlier in the stream, later chunks will shift to different batch indices. Your requirement (“same dataset → same batches unless a chunk is removed”) holds for identical datasets; if a chunk is removed, shifting is expected.
